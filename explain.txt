RTLSense: How It Was Built
A Technical Walkthrough for Students
Written by Claude (Anthropic) for Sandeep Ramlochan
=======================================================


WHAT WE BUILT
=============

RTLSense is a command-line tool that watches Verilog files and tells chip
designers — in under 20 seconds — whether their code will fail timing before
they ever run a real synthesis tool.

Think of it like this: a compiler tells you about syntax errors the moment you
save. RTLSense tells you about timing errors the moment you save. The insight
is that "roughly right in 15 seconds" beats "exactly right in 6 hours."

The tool has six layers. Each layer is a Python module. They are independent
and testable on their own, but they connect into a pipeline:

  File saved
      |
      v
  [watcher.py]       — detects which module changed
      |
      v
  [synthesizer.py]   — runs Yosys to convert RTL to gates
      |
      v
  [timing.py]        — runs OpenSTA to measure gate delays
      |
      v
  [mapper.py]        — maps timing violations back to source lines
      |
      v
  [explainer.py]     — asks Claude API to explain the problem in plain English
      |
      v
  [formatter.py]     — prints colored terminal output with the timing budget

Let's walk through each one.


-----------------------------------------------------------------------
BEFORE WE START: THREE TOOLS YOU NEED TO UNDERSTAND
-----------------------------------------------------------------------

Before touching any Python, you need to understand three external tools.
RTLSense does not do timing analysis itself — it orchestrates these three
tools and presents their output in a useful way. This is a critical design
decision. Never reinvent a wheel that a PhD already built.

1. YOSYS
   Yosys is an open-source logic synthesis tool. "Synthesis" means taking
   your Verilog (human-readable hardware description) and converting it into
   a netlist of actual logic gates — AND gates, OR gates, flip-flops — from
   a real chip library. Yosys was written by Claire Wolf and is the backbone
   of the entire open-source chip design ecosystem.

   You control Yosys by writing a small script (a .tcl or inline commands)
   and passing it to the yosys binary. Example:

       yosys -p "read_verilog my_module.v; synth -top my_module; write_verilog out.v"

   Yosys outputs statistics: how many gates, how many cells, logic depth.
   Most importantly, it preserves "src attributes" — tags on each internal
   wire that record which line of the original Verilog created it. This is
   the key that lets us map violations back to source lines.

2. OPENSTA
   OpenSTA is an open-source Static Timing Analysis tool from The OpenROAD
   Project. "Static Timing Analysis" (STA) means: given a synthesized
   netlist and a clock constraint, find every path from a register output
   to a register input, add up all the gate delays along that path, and
   report whether the total delay fits within the clock period.

   If your clock period is 2ns (500MHz) and a path takes 2.3ns, that path
   has a "slack" of -0.3ns. It will fail. The chip will not work at 500MHz.

   OpenSTA is controlled by a .tcl script. You tell it: read the liberty
   file (gate delay data), read the netlist, define the clock, report paths.
   It prints a detailed timing report that we parse in timing.py.

3. SKY130 LIBERTY FILE
   A liberty (.lib) file is a database of every standard cell in a chip
   library. It contains the delay of every gate under specific conditions
   (temperature, voltage, input slew). The SKY130 library is from Google
   and SkyWater Technology — it is a real, manufacturable 130nm process.
   Real chips have been taped out using this exact library.

   The liberty file is what makes RTLSense's numbers real. We are not
   guessing or using heuristics. We are using the actual delay data for
   actual silicon cells. That is the key differentiator from an AI predictor.


-----------------------------------------------------------------------
LAYER 0: config.py — LOAD SETTINGS FROM YAML AND ENV VARS
-----------------------------------------------------------------------

Build this first because every other module needs it.

The config module loads settings in priority order:
  1. Command-line flags (highest priority)
  2. rtlsense.yaml in the project root
  3. ~/.rtlsense.yaml in the home directory
  4. Hardcoded defaults (lowest priority)

Use Python's pyyaml library to parse YAML. Use os.environ to read
environment variables (for the Anthropic API key — never hardcode secrets).

Key config values:
  - pdk: which chip library to use (sky130 or asap7)
  - clock_period_ns: the target clock period
  - yosys_path: path to the yosys binary
  - sta_path: path to the OpenSTA binary
  - sky130_lib_path: path to the .lib file
  - wire_delay_factor: scaling factor for estimated wire delay (default 1.4)
  - warning_threshold_pct: warn if slack is less than 20% of clock period

Design tip: Return a dataclass or a simple object. Avoid passing around
raw dictionaries — it makes the code hard to read and hard to test.

Python skills used: pyyaml, os.environ, dataclasses, pathlib.Path


-----------------------------------------------------------------------
LAYER 1: watcher.py — DETECT WHICH MODULE CHANGED
-----------------------------------------------------------------------

The file watcher uses Python's watchdog library to monitor a directory
for changes. When a .v or .sv file is saved, it fires a callback.

The critical optimization here is: don't re-synthesize the whole design
every time. Only re-synthesize the module that changed. A large design
might have 50 modules. Changing one line should not require re-running
synthesis on all 50.

How to detect which module changed:
  1. Keep a cache of the last-seen content for each file.
  2. On change, diff the new content against the cache.
  3. Use a regex to extract module declarations from Verilog:
       r'module\s+(\w+)\s*[#(;]'
  4. For each module whose source lines changed, extract it into a
     temporary file for synthesis.

The cache is a simple Python dictionary: {filename: content_string}.
Persist it between runs using Python's pickle or shelve module so you
don't re-synthesize on startup.

Python skills used: watchdog, re (regex), hashlib (file hashing),
                    tempfile, shelve


-----------------------------------------------------------------------
LAYER 2: synthesizer.py — WRAP YOSYS IN PYTHON
-----------------------------------------------------------------------

This is the most critical module. Get this right before anything else.

The synthesizer's job:
  1. Take a Verilog module as input
  2. Write a Yosys synthesis script
  3. Run Yosys as a subprocess
  4. Parse Yosys's stdout for statistics
  5. Return the synthesized netlist path + statistics

How to run Yosys from Python:

    import subprocess
    result = subprocess.run(
        ['yosys', '-p', yosys_script],
        capture_output=True,
        text=True,
        timeout=60
    )

The Yosys script (passed as a string to -p) looks like this:

    read_verilog -sv {input_file}
    hierarchy -top {module_name}
    proc
    opt
    fsm
    opt
    memory
    opt
    techmap
    abc -liberty {sky130_lib_path}
    stat
    write_verilog -attr2comment {output_netlist}

CRITICAL DETAIL — src attribute preservation:
When you run "proc" and "opt", Yosys moves logic around but tries to
keep "src" attributes (source file + line number) on internal wires.
The "-attr2comment" flag on write_verilog writes these as comments in
the output netlist, like:
    /* src="game_controller.v:47" */
We parse these comments in mapper.py to trace violations back to lines.

What to parse from Yosys stdout:
  - Number of cells: look for "Number of cells:" in the stat output
  - Cell area: Yosys reports total cell area from the liberty file
  - Logic levels: abc reports "ABC: Delay = X.XX" — this is your logic depth
  - Synthesis errors: if returncode != 0, parse stderr for the error message
    and return it as a user-friendly diagnostic instead of crashing

Python skills used: subprocess, re, tempfile, dataclasses, logging


-----------------------------------------------------------------------
LAYER 3: timing.py — WRAP OPENSTA IN PYTHON
-----------------------------------------------------------------------

The timing analyzer runs OpenSTA and parses its timing report. This is
two sub-problems: (1) running OpenSTA correctly, (2) parsing its output.

Part A: Running OpenSTA

Write a temporary .tcl script:

    read_liberty {pdk_liberty_file}
    read_verilog {synthesized_netlist}
    link_design {module_name}
    create_clock -period {clock_period} [get_ports clk]
    set_input_delay {input_delay} -clock clk [all_inputs]
    set_output_delay {output_delay} -clock clk [all_outputs]
    report_checks -path_delay max -format full_clock_expanded \
        -fields {capacitance slew input_pins nets fanout} -digits 4
    report_checks -path_delay max -endpoint_count 20
    report_tns
    report_wns
    exit

Then run: subprocess.run(['sta', tcl_script_path], ...)

THE WIRE DELAY TRAP — read this carefully:
At this stage we have NO routing information. Yosys only knows about
gates, not wires. In a real chip, wires add significant delay — often
30-50% of total path delay at 130nm. Without wire delays, OpenSTA will
show paths that appear to meet timing but will actually fail after
place-and-route. This is a false negative — the worst kind.

Solution: apply a wire_delay_factor (default 1.4) to all reported path
delays. If OpenSTA reports 1.4ns, we report 1.4 * 1.4 = 1.96ns.
This is disclosed to the user in the output. It is not hiding a flaw —
it is a deliberate, documented engineering decision to prevent false
confidence. The config makes it adjustable for users who know better.

Part B: Parsing the OpenSTA timing report

OpenSTA's report_checks output looks like this (simplified):

    Startpoint: reg_a (rising edge-triggered flip-flop clocked by clk)
    Endpoint: reg_b (rising edge-triggered flip-flop clocked by clk)
    Path Group: clk
    Path Type: max

      Delay    Time   Description
    ---------------------------------------------------------
       0.00    0.00   clock clk (rise edge)
       0.12    0.12   ^ reg_a/CK (sky130_fd_sc_hd__dfxtp_1)
       0.35    0.47   ^ and2/A (sky130_fd_sc_hd__and2_1)
       0.18    0.65   ^ and2/X (sky130_fd_sc_hd__and2_1)
    ...
       1.67    1.67   data arrival time

       2.00    2.00   clock period
      -0.20   -0.20   output delay
       1.80           data required time
    ---------------------------------------------------------
      1.80           data required time
      1.67           data arrival time
      0.13           slack (MET)

Write a parser using Python's re module. Key things to extract:
  - Startpoint name and type
  - Endpoint name and type
  - Each cell in the path: name, cell type, delay, cumulative time
  - Path arrival time (total gate delay)
  - Required time (clock period minus constraints)
  - Slack (required - arrival; negative means violation)
  - Count of logic cells in path (this is "logic depth")

Also parse report_wns (worst negative slack) and report_tns (total
negative slack, sum of all violations).

Apply wire_delay_factor to the reported arrival time before computing
the adjusted slack.

Python skills used: re, subprocess, dataclasses, tempfile, logging


-----------------------------------------------------------------------
LAYER 4: mapper.py — CONNECT VIOLATIONS TO SOURCE LINES
-----------------------------------------------------------------------

The mapper answers the question: "Which line of my Verilog caused this?"

Two methods, used in priority order:

METHOD 1: src attribute parsing (primary, works for registers)
When Yosys writes the netlist with -attr2comment, comments appear like:
    /* src="game_controller.v:47" */
Parse these with regex. Build a dictionary:
    {signal_name: (filename, line_number)}

A timing path has a Startpoint (a register) and an Endpoint (a register).
These usually survive synthesis with their original names. Look up the
startpoint and endpoint in your src dictionary. This gives you the
exact Verilog line of the flip-flop that starts or ends the path.

METHOD 2: signal name grep (fallback, works when method 1 fails)
The timing path lists intermediate signal names like "collision_detected".
Open the original Verilog file and grep for where that signal is assigned:
    re.search(r'assign\s+collision_detected|collision_detected\s*<=', ...)
This is imperfect — if the optimizer renamed the signal, it won't work.
But for a demo with controlled examples, it works well enough.

Output for each violation:
  - filename
  - line_number (best estimate)
  - signal_name
  - path_delay (adjusted for wire factor)
  - slack
  - logic_depth (cell count in path)
  - cells_in_path (list of cell names)
  - code_snippet (5 lines from the original file centered on line_number)

The code_snippet is extracted by reading the original .v file and slicing
lines[line_number-2 : line_number+3]. This is shown in the terminal output
next to the violation so the designer sees the actual code immediately.

Python skills used: re, pathlib, dataclasses


-----------------------------------------------------------------------
LAYER 5: explainer.py — ASK CLAUDE TO EXPLAIN THE VIOLATION
-----------------------------------------------------------------------

This layer takes a timing violation and returns a 2-3 sentence explanation
in plain engineer language, plus a concrete fix suggestion.

It calls the Anthropic API (claude-sonnet-4-20250514) with:
  - A system prompt describing the AI's role as a senior ASIC engineer
  - A user message containing: the code snippet, path delay, slack,
    logic depth, cells in path, wire delay estimate

The system prompt is carefully written to produce short, specific answers:
  "In 2-3 sentences, explain WHY this code causes a timing problem.
   Be specific about the logic structure (carry chains, wide muxes, etc.)
   Suggest ONE concrete RTL fix. Reference actual signal names."

Caching is critical here for two reasons:
  1. API calls cost money
  2. The same violation will appear many times as the designer iterates
     (fix the slack a little, save again, same fundamental problem)

Cache key: hash of (code_snippet + cells_in_path). Store in a JSON file
in .rtlsense_cache/explanations.json. Check the cache before every API call.

The --no-ai flag skips this layer entirely. Make the rest of the tool
work without it — the AI explanation is a nice-to-have, not load-bearing.

Python skills used: anthropic SDK, hashlib, json, functools.lru_cache


-----------------------------------------------------------------------
LAYER 6: formatter.py — PRINT BEAUTIFUL TERMINAL OUTPUT
-----------------------------------------------------------------------

This is what the user sees. It needs to look professional.

Use Python's Rich library. Rich gives you:
  - Colors and styles (red for violations, yellow for warnings, green for met)
  - Tables, panels, progress bars
  - Syntax highlighting for code snippets

For each violation, print a panel containing:
  - The violation header (red): "TIMING VIOLATION: game_controller.v:47"
  - The code snippet with the offending line highlighted in red
  - A metrics row: Slack: -0.47ns | Logic Depth: 23 levels | Path: 1.96ns
  - The timing budget visualizer (described below)
  - The AI suggestion (if available)

THE TIMING BUDGET VISUALIZER:
Show how the clock period is consumed as a simple bar:

    Clock Budget [████████████████░░░░] 1.96ns / 2.00ns
                  gates: 1.40ns   wire est: 0.56ns   margin: 0.04ns

Implement this with Rich's Progress or by building a string manually.
The math: bar_fill = int((path_delay / clock_period) * 20) characters.

Summary line at the bottom (always visible):
  "3 violations | 2 warnings | 12 paths met | WNS: -0.47ns | TNS: -1.20ns"

JSON output mode (--json flag):
  Output a list of diagnostic objects compatible with VS Code's diagnostic
  format. Each object has: file, line, column, severity, message,
  path_delay, slack, logic_depth, suggestion. This makes future editor
  integration trivial.

Python skills used: rich (Console, Panel, Text, Progress), json, dataclasses


-----------------------------------------------------------------------
LAYER 7: cli.py — THE ENTRY POINT
-----------------------------------------------------------------------

The CLI is the glue that connects all layers. Use Python's Click library.

Click uses decorators to define commands and options:

    @click.group()
    def rtlsense():
        pass

    @rtlsense.command()
    @click.argument('verilog_file')
    @click.option('--clock', required=True, help='Clock frequency e.g. 500MHz')
    @click.option('--pdk', default='sky130')
    @click.option('--no-ai', is_flag=True)
    @click.option('--json', 'json_output', is_flag=True)
    @click.option('--verbose', is_flag=True)
    def check(verilog_file, clock, pdk, no_ai, json_output, verbose):
        config = load_config(clock=clock, pdk=pdk)
        # run the pipeline: synthesize → time → map → explain → format
        ...

Parse the clock frequency string ("500MHz", "1GHz", "2.4GHz") into a
period in nanoseconds. This is a simple string operation:
  - Strip non-numeric suffix to get value
  - Convert unit (MHz → ns = 1000/MHz, GHz → ns = 1/GHz)

The 'watch' command wraps the 'check' logic in a watchdog loop.
The 'diff' command runs 'check' twice and compares the results.

Python skills used: click, logging


-----------------------------------------------------------------------
THE DIFF MODE — HOW TO COMPARE TWO VERSIONS
-----------------------------------------------------------------------

Diff mode is the "killer feature." It answers: "Did my change make this worse?"

Algorithm:
  1. Synthesize and time version 1. Store results as dict keyed by
     (startpoint, endpoint): TimingPath object.
  2. Synthesize and time version 2. Store results the same way.
  3. For each path in version 2, look it up in version 1 by key.
     - If the path exists in both: compute delta_slack = v2.slack - v1.slack
     - If the path is new in v2: it's a new violation (bad)
     - If the path disappeared from v2: it was fixed (good)
  4. Sort by delta_slack (worst regression first).
  5. Print a comparison table: path name, v1 slack, v2 slack, delta, verdict.

The summary line is what makes this feature:
  "Your change made WNS 0.47ns worse. 2 new violations introduced.
   Logic depth increased from 8 to 23 on the collision_detected path."

This is what a designer reads at 2am when they're not sure if their last
edit helped or hurt. A single clear sentence. That is the product.


-----------------------------------------------------------------------
EXAMPLE VERILOG FILES — WHY THEY MATTER
-----------------------------------------------------------------------

The example files are not an afterthought. They are the demo. They need
to be carefully designed so:
  1. They synthesize in under 15 seconds (keep them small, under 2000 lines)
  2. The "bad" version clearly fails timing (logic depth should jump visibly)
  3. The signal names are intuitive so the AI explanation makes sense

riscv_alu.v: A clean 16-bit ALU. Operations: ADD, SUB, AND, OR, XOR, SLT.
  All paths should meet 500MHz comfortably. Logic depth around 8 levels.
  This is your baseline. It should look green.

riscv_alu_bad.v: Same ALU, but 64-bit with a ripple carry adder and a
  priority encoder feeding into the result mux. Logic depth jumps to 25+.
  The priority encoder creates a wide, deep combinational cone. This is
  exactly the kind of thing a junior designer writes by accident.

game_controller_v1.v: A Tetris game controller with pipelined collision
  detection. All paths meet 500MHz. Good RTL hygiene: intermediate
  registers break long paths.

game_controller_v2.v: Same controller, three intentional regressions:
  - Removed pipeline registers from collision detection
  - Added an 8-bit multiplier in the scoring logic (deep carry chain)
  - Changed FSM from one-hot to binary (deeper state decode)

The three-act demo script (demo.sh) runs these in sequence and shows
the timing numbers changing. This is what you run in front of the CEO.


-----------------------------------------------------------------------
SETUP AND INSTALLATION
-----------------------------------------------------------------------

setup.sh does five things:
  1. Checks if yosys is installed (brew install yosys)
  2. Checks if sta is installed (build OpenSTA from source — no apt package)
  3. Downloads the SKY130 liberty file from GitHub if not present
  4. Validates the install by synthesizing a tiny test (a single AND gate)
  5. Runs a tiny STA to confirm timing output is produced

The validation step is critical. It catches misconfiguration early and
gives a clear error message instead of a cryptic crash during the demo.

Pre-downloading the liberty file is critical for demos. The sky130 .lib
file is ~15MB. You do not want a download happening during a live demo.


-----------------------------------------------------------------------
TESTING STRATEGY
-----------------------------------------------------------------------

Test the parsers, not the tools. You don't need Yosys installed to test
the Yosys output parser. Write the parser against a captured sample of
real Yosys stdout (check it into the tests/ directory as a fixture).

Same for the OpenSTA parser: capture a real timing report, save it as
tests/fixtures/timing_report.txt, write a test that parses it and
checks that WNS, TNS, path count, and slack values are extracted correctly.

tests/test_synthesizer.py  — test the output parser, not the subprocess
tests/test_timing_parser.py — test the STA report parser
tests/test_mapper.py       — test src attribute extraction from a sample netlist

Use pytest. Keep tests fast (no subprocess calls, no tool dependencies).


-----------------------------------------------------------------------
KEY ENGINEERING DECISIONS AND WHY WE MADE THEM
-----------------------------------------------------------------------

1. WHY YOSYS + OPENSTA INSTEAD OF JUST ESTIMATING?
   Because "real numbers" is the product. If we used heuristics or ML,
   we'd be building a predictor. Predictors are wrong in non-obvious ways.
   Yosys + OpenSTA are deterministic. Wrong in obvious ways. Trust matters
   more than accuracy for a shift-left tool.

2. WHY MODULE-LEVEL ISOLATION?
   Speed. Synthesizing one module in 10 seconds is useful. Synthesizing
   a 500-module design in 8 hours is not. The tradeoff is that cross-module
   paths are invisible. We accept this and document it.

3. WHY SKY130 INSTEAD OF A PREDICTIVE PDK?
   SKY130 is real silicon. Real tapeouts. Real numbers. If someone builds
   a chip and says "RTLSense said this path would fail, and it did" — that
   is a testimonial. ASAP7 is predictive. Good for 7nm intuition,
   but no real chips have been taped out using it.

4. WHY THE WIRE DELAY FACTOR?
   Without it, synthesis-only timing is dangerous. A path that shows
   +0.1ns slack with gates only might fail with -0.3ns after routing.
   The 1.4x factor prevents false confidence. The exact value is less
   important than the principle: never let pre-layout timing look cleaner
   than post-layout reality.

5. WHY CLI INSTEAD OF A WEB APP OR VS CODE EXTENSION?
   CLI is faster to build and easier to demo. The VS Code extension is
   the product vision — a future step. Trying to build it in one weekend
   guarantees you ship nothing. Ship the CLI. Prove the concept. Extend later.

6. WHY CLAUDE API FOR EXPLANATIONS?
   Because "you have a 23-level carry chain" is a diagnosis. "Pipeline
   the adder by registering the intermediate carry at bit 32" is a fix.
   The timing tool gives you the diagnosis. The AI gives you the fix.
   Those are different cognitive tasks. Let each tool do what it's best at.


-----------------------------------------------------------------------
WHAT WE DID NOT BUILD AND WHY
-----------------------------------------------------------------------

- No VS Code extension: would take weeks, not a weekend
- No web UI: CLI is the right first interface for engineers
- No ML-based prediction: the value is determinism, not ML
- No placement/routing: Yosys + STA only, and we disclose this
- No cross-module analysis: too slow, out of scope for v1
- No full SystemVerilog support: Yosys handles a subset; we document limits
- No sign-off accuracy claims: we are a development aid, not a signoff tool


-----------------------------------------------------------------------
HOW TO READ THIS AS A STUDENT
-----------------------------------------------------------------------

If you are learning to build tools like this, the most important lessons are:

1. ORCHESTRATE, DON'T REINVENT.
   Yosys took years to build. OpenSTA took years to build. RTLSense took
   a weekend because it stands on those shoulders. Find the best existing
   tools for each sub-problem and connect them.

2. PARSE REAL OUTPUT, NOT FAKE OUTPUT.
   Run Yosys. Run OpenSTA. Capture what they actually print. Write your
   parser against that. Do not guess the format. Do not trust documentation
   over observed behavior.

3. THE PIPELINE ARCHITECTURE IS A GIFT.
   Because each layer (synthesizer, timing, mapper, formatter) has a clean
   input and output, you can test each one in isolation. You can swap out
   Yosys for a different synthesis tool without touching the timing layer.
   You can change the terminal formatting without touching the parser.
   Design your software so the pieces can breathe.

4. THE DEMO IS THE PRODUCT.
   The Three-Act Demo exists because a working demo is more persuasive than
   any README. When you build tools, design your examples to be demonstrable.
   Small enough to run fast. Different enough to show a clear before/after.
   Named well enough that a non-expert understands what changed.

5. HONEST LIMITATIONS BUILD TRUST.
   RTLSense cannot handle cross-module paths. It uses 130nm data for 5nm
   designs. Its wire delay estimate is a rough factor. We say all of this
   in the README. A CEO with a PhD knows these limitations exist. If you
   don't name them, she'll think you don't know about them. If you name
   them, she'll think you're trustworthy.
